# OOP(객체지향프로그래밍)

### 📌 브랜치 규칙 및 네이밍 (Branch Rules & Naming)

1. 모든 기능 개발은 **자신의** 브랜치에서 시작
2. 작업 시작 전, 항상 최신 `자신의 브랜치` 내용 받아오기 (`git pull origin "브랜치 이름"`)
3. 작업 완료 후, `자신의 브랜치`로 Pull Request(PR) 생성(중요!! 다른 사람브랜치로 하면 안됨..!)
4. PR에 학습 내용 간단히 작성 후 머지

### 🎯 커밋 컨벤션 (Commit Convention)

- **주의 사항:**
- `type`은 소문자만 사용 (feat, fix, refactor, style, test, chore)
- `subject`는 **모두 현재형 동사**
- 기능 개발할때마다 커밋하는 습관 가지기!

| type | subject |
| --- | --- |
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| style | 코드 포맷팅, 오타 수정, 주석 수정 및 삭제 등 |
| refactor | 코드 리팩토링 |
| test | 테스트 코드 |
| chore | 빌드 및 패키지 수정 및 삭제 |
| merge | 브랜치를 머지 |

```bash
#### ✨ 예시
feat: 리뷰 작성 api 추가
fix: 로그인 오류 해결
```

### Issue 템플릿

- `feat`
    - 팀원 모두 주차 별로 구현해야하는 기능에 대한 이슈 작성 (작업 및 변경 사항 확인 목적)
    - to-do에 구현해야할 기능을 작성하고, 구현이 끝나면 체크표시

- `bug`
    - 구현 중 해결하지 못하는 버그 발생 시 작성
